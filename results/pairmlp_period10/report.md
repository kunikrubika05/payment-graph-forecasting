# PairwiseMLP — Эксперименты E1–E4 (period_10)

**Дата:** 2026-03-31
**Ветка:** `mlp`
**Артефакты:** `orbitaal_processed/experiments/exp_pairmlp_10_E{1..4}` (Яндекс.Диск)

---

## 1. Задача и мотивация

Задача — **temporal link prediction**: для каждого ребра (src, dst) из val/test периода предсказать вероятность его существования в ранжировании среди 100 негативных кандидатов. Протокол оценки идентичен TGB (tgbl-coin): per-source ranking, метрика MRR (primary), H@1, H@3, H@10.

PairwiseMLP — легковесная модель, которая использует исключительно **структурные признаки пары (src, dst)**, вычисленные из train-части графа. Никаких node embeddings, истории транзакций, временных меток — только топологические признаки из adjacency матрицы.

Мотивация: литература показывает, что Common Neighbors — очень сильный предиктор для Bitcoin-сети (MRR=0.6403 на нашем датасете). Вопрос: может ли обученный MLP поверх структурных признаков превзойти простую эвристику?

---

## 2. Датасет и разбивка

**Датасет:** ORBITAAL — Bitcoin entity-level транзакционный граф, период `2020-06-01 – 2020-08-31`.
**Period_10:** первые **10% рёбер** полного stream graph по времени.

| Раздел | Рёбра | Доля |
|--------|-------|------|
| Train  | 4,300,506 | 70% |
| Val    | 921,537   | 15% |
| Test   | 921,538   | 15% |
| **Всего** | **6,143,581** | — |

**Граф (из train):** 1,933,429 активных нод, directed nnz = 3,749,871, undirected nnz = 7,427,636.

Разбивка **строго хронологическая**: train = первые 70% рёбер по timestamp, val = следующие 15%, test = последние 15%. Нет перекрытия по времени.

Полный stream graph содержит 61,435,816 рёбер; period_10 — это самое начало периода (данные за ~9 дней из 92 дней окна, т.к. активность неравномерна).

## 3. Предвычисленные артефакты (offline precompute)

Тяжёлые вычисления вынесены в отдельный шаг (`src/models/pairwise_mlp/precompute.py`), который запускается один раз на CPU-машине и загружает результаты на Яндекс.Диск. GPU-машина скачивает готовые артефакты при старте.

### Что предвычисляется

**Шаг 1–3 (загрузка данных):** stream graph, adjacency матрицы (`adj_10_directed.npz`, `adj_10_undirected.npz`), node mapping (`node_mapping_10.npy`).

**Шаг 4 — negative sampling для train (k_neg_train=20):**
Для каждого из 4,300,506 train-рёбер семплируется 20 негативных dst:
- до 5 исторических негативов (соседи src из train-части, отсутствующие в конкретном batch-окне)
- остальные — случайные из активных нод

Результат: `neg_dst.npy` (4,300,506 × 20, int64). Время: ~47 мин на CPU (однопоточно, ограничение — итерация по 4.3M источников).

**Шаг 5 — предвычисление признаков:**
Для каждого positive-ребра и для каждой из 20 его neg-пар вычисляются 7 структурных признаков.

- `pos_features.npy`: (4,300,506, 7), float32 — признаки positive пар
- `neg_features.npy`: (4,300,506, 20, 7), float32 ≈ **2.4 GB** — признаки negative пар

Оба файла загружаются в RAM GPU-машины при старте. Это позволяет не пересчитывать признаки в каждую эпоху — только GPU forward pass.

**Важная техническая деталь:** scipy CSR fancy indexing (`adj[vs]`) внутренне вызывает `sort_indices()`, что модифицирует матрицу in-place → не thread-safe. Параллельная версия с `joblib threading` вызывала segfault. Решение: `adj.sort_indices()` перед вычислением + однопоточный `compute_features_batch`.

## 4. Признаки пары (src, dst)

Все 7 признаков вычисляются **только из train-adjacency** (нет data leakage).
Ноды, отсутствующие в train → все признаки = 0.

| Индекс | Название | Описание |
|--------|----------|----------|
| 0 | `cn_uu` | Common Neighbors, undirected (целое ≥ 0) |
| 1 | `log1p_cn_uu` | log(1 + cn_uu) |
| 2 | `aa_uu` | Adamic-Adar undirected: ∑_{w ∈ N(u)∩N(v)} 1/log(deg(w)) |
| 3 | `cn_dir` | Common Neighbors, directed: |out-N(src) ∩ out-N(dst)| |
| 4 | `aa_dir` | Adamic-Adar directed |
| 5 | `jaccard_uu` | Jaccard undirected: CN / (deg(u) + deg(v) − CN) ∈ [0,1] |
| 6 | `log1p_pa_uu` | log(1 + deg_src × deg_dst), undirected (Preferential Attachment) |

**Диапазоны:** cn_uu ∈ [0, 1000+], jaccard ∈ [0, 1], log1p_pa_uu ∈ [0, 15+] — большой разброс масштабов, что делает BatchNorm на входе необходимым.

---

## 5. Архитектура модели (PairMLP)

```
BatchNorm1d(n_features)
→ Linear(n_features, 64) → ReLU
→ Linear(64, 32) → ReLU
→ Linear(32, 1)
```

Выход — скалярный score (чем выше, тем вероятнее ребро). BatchNorm на входе нормализует разнородные шкалы признаков. При `dropout > 0` слои Dropout добавляются после каждого ReLU.

| Конфиг | Параметры |
|--------|-----------|
| 1 → [64,32] → 1 | 2,243 |
| 3 → [64,32] → 1 | 2,375 |
| 7 → [64,32] → 1 | 2,639 |

## 6. Протокол обучения

**Optimiser:** Adam, lr=0.001, weight_decay=1e-4, grad_clip=1.0
**Batch size:** 4096 позитивных пар → 1050 батчей/эпоха
**Epochs:** max 50, patience=10 eval checks (early stopping по val MRR)
**Eval every:** 4 эпохи

### Функции потерь

**BPR (Bayesian Personalized Ranking):**
`L = −mean(log σ(score_pos − score_neg))` по всем (pos, neg) парам.
Напрямую оптимизирует ранжирование → aligned с MRR.

**BCE (Binary Cross-Entropy):**
`L = ½[BCE(score_pos, 1) + BCE(score_neg, 0)]`
Стандартная бинарная классификация; менее прямолинейно связана с MRR.

### Стратегия негативов при обучении

**Precomputed fixed negatives + k_neg_sample:**
Негативы предвычислены один раз (k_neg_train=20 на positive). Если использовать одни и те же 20 негативов в каждую эпоху, модель быстро их запоминает → деградация (H@1=H@3=H@10 ≈ 0.54 → step function).

Решение: параметр `k_neg_sample=10` — каждый батч случайно выбирает 10 из 20 precomputed негативов через `torch.randperm`. Модель видит разные подмножества в каждую эпоху, что действует как data augmentation без пересчёта признаков.

## 7. Протокол оценки (TGB-style)

Идентичен протоколу sg_baselines. Полное описание в `docs/evaluation_protocols_temporal_lp_ru.md`.

**Per-source ranking:**
Для каждого positive ребра (s, d_true) из val/test:
1. Строится candidate set: {d_true} ∪ {100 негативных dst}
2. Модель ранжирует 101 кандидата по score
3. Фиксируется ранг d_true → вычисляются MRR, H@1, H@3, H@10

**Отбор запросов:**
- Отфильтровываются ребра, где src или dst не встречались в train (нет признаков = бессмысленный запрос)
- Из оставшихся случайно выбирается до 50,000 запросов (subsampling с fixed seed)
  - Val: 162,645 → 50,000 (seed=52)
  - Test: 108,840 → 50,000 (seed=62)

**Кэширование eval:** негативы и признаки для val/test вычисляются один раз перед обучением (~10 мин, одноразово), затем каждый eval = только GPU forward pass (~0.4–0.6с).

**Референсные значения эвристик (test):**

| Эвристика | MRR | H@1 | H@3 | H@10 |
|-----------|-----|-----|-----|------|
| Common Neighbors | 0.6403 | — | — | — |
| Adamic-Adar | 0.6255 | — | — | — |

## 9. Детальный анализ по экспериментам

### E1: cn_uu + BPR — лучший результат

Единственный признак — `cn_uu` (число общих соседей, undirected). Одно число однозначно описывает структурную близость пары.

**Динамика обучения:**
```
Epoch  4: val_MRR=0.6771 ← best
Epoch  8: val_MRR=0.5440 (деградация: H@1=H@3=H@10=0.5390)
Epoch 12: val_MRR=0.6768 (восстановление)
Epoch 16–44: осцилляция между 0.54 и 0.55 (step function)
```

Паттерн: модель нестабильна. Чётные эпохи (когда eval происходит) дают разные результаты из-за случайного subsampling негативов. При val_MRR=0.5440 картина `H@1=H@3=H@10` — это **step function**: модель присваивает всем кандидатам одинаковые scores, и ранг d_true ≈ 50-й из 101 (случайный). Затем на следующих eval-точках модель выходит из этого состояния.

Лучший чекпоинт (epoch 4) сохранён → на нём Test MRR = 0.6026.

**Вывод:** val MRR = 0.6771 > CN(0.6403), то есть на val модель **превзошла эвристику**. Но на test MRR = 0.6026 < 0.6403 — хуже CN. Это указывает на temporal distribution shift: train adjacency менее информативна для более поздних рёбер (test период).

### E2: cn_uu + log1p_cn_uu + aa_uu + BPR — хуже E1

Добавление `log1p_cn_uu` и `aa_uu` к `cn_uu` снизило результат: test MRR 0.6026 → 0.5721.

**Динамика:** loss упал до ~0.001 уже к эпохе 2 (model memorised train negatives?), затем застрял. Val MRR осциллировал между 0.597 и 0.653. Лучший checkpoint = epoch 8 (0.6535 val).

**Причина деградации:** `log1p_cn_uu` — монотонная трансформация `cn_uu`, добавляет мало новой информации, но усложняет пространство решений. `aa_uu` коррелирует с `cn_uu`. BatchNorm может плохо обрабатывать коллинеарные признаки. В итоге модель хуже обобщается на test.

### E3: все 7 признаков + BPR — катастрофа

Test MRR = 0.3075 — почти вдвое хуже CN эвристики. Val MRR не поднялся выше 0.34.

**Причина:** добавление `log1p_pa_uu` (Preferential Attachment = log(deg_src × deg_dst)) и `jaccard_uu`. PA — опасный признак: у хабов (высокая степень) PA очень большой для любой пары, где один из узлов — хаб. Поскольку random negatives тоже могут включать хабы (с вероятностью пропорциональной степени в uniform sampling), модель обучается давать высокий score парам с высоким PA, что разрушает ранжирование для большинства non-hub запросов.

Jaccard ∈ [0,1] ортогонален по шкале к остальным, что вместе с PA создаёт трудноразделимое пространство. BN + [64,32] MLP недостаточно мощен, чтобы правильно взвесить 7 признаков с такими свойствами.

### E4: все 7 признаков + BCE — немного лучше E3, но всё равно плохо

Test MRR = 0.3261, val MRR = 0.3747 (epoch 32). BCE чуть стабильнее (сходится к epoch 32 против epoch 4 в E3), но проблема с PA/jaccard признаками остаётся.

**Наблюдение:** BCE training показывает высокую дисперсию — val MRR от 0.17 до 0.37 в разные эпохи, без монотонного роста. Это нетипично и указывает, что 7-признаковое пространство с данной архитектурой плохо оптимизируется.

## 10. Ключевые наблюдения

### 10.1 Меньше признаков — лучше

| Признаков | Test MRR |
|-----------|---------|
| 1 (cn_uu) | 0.6026 |
| 3 (+ log1p_cn + aa_uu) | 0.5721 |
| 7 (все) | 0.3075–0.3261 |

Простейшая модель выигрывает. Это согласуется с литературой (Bianconi & Agrawal, 2017): для Bitcoin-сети простые структурные метрики не уступают сложным комбинациям.

### 10.2 PA (Preferential Attachment) — вредоносный признак

`log1p_pa_uu` сильно ухудшил результат в E3/E4. PA смешивает информацию о степени узлов с информацией о структуре пары, что при нашей стратегии негативного сэмплинга (частично random) создаёт артефакты: модель учится использовать степень как прокси популярности, что не помогает ранжированию.

### 10.3 Модель не побила CN эвристику на test

E1: test MRR = 0.6026 vs CN = 0.6403. Разрыв: -0.0377.

Теоретически MLP с единственным признаком cn_uu должна **совпадать** с CN эвристикой (она является монотонной функцией cn_uu). Почему разрыв?
1. **Temporal shift:** train adjacency вычислена на первых 70% рёбер, test — последние 15%. За это время граф эволюционирует, новые связи появляются у узлов, чьи cn_uu низкое в train периоде.
2. **BatchNorm на inference:** BN использует batch statistics, которые могут отличаться между train и test распределениями.
3. **Best checkpoint = epoch 4** (очень ранняя остановка): модель не успела полностью converge.

### 10.4 Нестабильность обучения (E1/E2)

Деградация epoch 4→8 (0.6771→0.5440) в E1 повторяется. Несмотря на k_neg_sample=10, модель периодически попадает в step-function режим. Возможная причина: при eval_every=4 между eval-точками модель делает 4×1050 = 4200 шагов без обратной связи. BatchNorm stats могут дрейфовать. Возможное решение: eval_every=2 или LR scheduler (cosine/reduce-on-plateau).

## 11. Выводы

**Что работает:**
- PairwiseMLP с одним признаком cn_uu + BPR даёт **val MRR=0.677** — это выше CN (0.640) на val, и является конкурентоспособным результатом среди методов без node embeddings.
- Архитектура и pipeline работают корректно. Нет data leakage (признаки только из train).
- Eval caching + k_neg_sample = правильный протокол.

**Что не работает:**
- Добавление большего числа признаков ухудшает результат. PA в особенности опасен.
- Test MRR = 0.6026 < CN = 0.6403 — модель не превзошла простую эвристику на test.
- Нестабильность обучения (осцилляция val MRR) не устранена k_neg_sample=10.

**Открытые вопросы для продолжения экспериментов:**
1. Нужен ли LR scheduler (cosine decay или ReduceLROnPlateau) для стабилизации?
2. Как поведёт себя модель на period_25 (больше данных → более стабильные признаки)?
3. Помогут ли node features (15 признаков из `features_10.parquet`) в дополнение к cn_uu?
   Тогда вход: 15 (node_src) + 15 (node_dst) + 1 (cn_uu) = 31 признак.
4. Можно ли стабилизировать обучение более агрессивным k_neg_sample (например, k=5 из 20)?
